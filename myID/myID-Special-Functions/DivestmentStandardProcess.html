Authoratative Source

  • Create a new application in myID with appropriate connector for the Authoratative source of the Divestment users.
  • The identities will be created after the aggregation from this source.
  • In past, for Russia divestment project we used our existing PMD application, so we created a new application with appropriate query to aggregate the users and create identities.
  • For Sherwin divestment project we used a CSV file as users source, so we created a new application with delimited connector. We used basic attributes of the users from PMD and added them in the CSV as source.

Identity creation process

  • We need to use ‘Create Identity Rule’ or ‘Build Map Rule’ (in case of JDBC or delimitor) to create the divestment identities correctly in order to separate them from normal identities.
  • Every divestment identity must have identity type as ‘Divestment’ and sub type as ‘Personal Div’
  • We have to cautious as the divestment users already have an identity in myID and we are now creating a new divestment identity for these users with almost similar attributes. To differentiate these users we have a different identity type, but other unique attributes like Personid, upn, windows login id as well needs to be adjusted.
  • Please use below process to change these unique attributes.
  • For Personid and Windows Login ID: Add prefix ‘div_’ to both of these attribute values.

For example, if personid of the user is 20304050, then for the divestment identity the personid should be ‘div_20304050’. Similarly with Windows Login ID, if the value is ‘bhakta’, then for divestment identity the value should be ‘div_bhakta’.

For UPN: The postfix of the UPN ‘@henkel.com’ should be changed to ‘@div.henkel.com’ for all users, the prefix can be kept same as the existing identity.

 

 Windows AD Account Creation

  • If there is a requirement to create AD accounts for the divestment users at the time of identity creation, then first align with the AD team.
  • The AD team should create a new OU in AD and we must create the accounts in that OU. We must not use any existing OU for the accounts creation.
  • The OU creation part and any access related things will be done by the AD team.
  • For AD accounts, create a new Active Directory application in myID and use the newly created OU in the filters and configure the application.
  • For accounts creation either use ‘Batch Request’ or create a business role with AD provisioning policy and assign the role to user. (**Batch request sometime do not create correct AD accounts)
  • While creating the account also create password for the Windows account using rule ‘Divestment-SimpleADPwdGenerator’ in the provisioning policy for password attribute.
  • Later these passwords will be shared with the users to login via secure channels by project responsible person.

Other accounts creation

  • If any other accounts should be created for the divestment users, then similar approach should be used like Windows AD accounts.
  • For passwords of these accounts, we should use the same password which was generated for AD accounts (refer to point 3 for the rule).

 Entitlement Assignment

  • Normally the divestment users will not have any default entitlements assigned at the time of account creation.
  • If any entitlements are needed to be added to these users then we have to use Sailpoint Roles. For this reason we have a new type of role created in myID called ‘Divestment’.
  • We should create a Divestment role which will be assigned and an IT role with the required entitlement in the Divestment role.
  • We can either use Batch Request or normal access request to assign these roles. (Only service desk users can request the access from myID. Appropriate access controls are already in place)

 Access Control of Divestment Identities

  • We have the access controls in place for the Divestment identities, provided the identity type is ‘Divestment’.
  • The divestment users cannot be searched on manage access and manage account pages.
  • Only Service desk users can search the divestment users and request for any access.
  • We have a separate role type of divestment users which can only be assigned to divestment users.

 Cutoff date activities

  • Disable AD accounts for all normal identities.
  • For divestment users, if any AD accounts are disabled, activate them.
  • If any of the divestment users cannot login into AD, reset the password using Batch Request and provide new password for first time login.
